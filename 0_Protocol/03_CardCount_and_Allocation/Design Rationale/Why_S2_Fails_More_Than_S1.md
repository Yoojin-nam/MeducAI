# 왜 S1보다 S2에서 더 자주 실패하는가

*(MeducAI Pipeline – Design Rationale & Operational Insight)*

## 1. 관찰 요약 (Empirical Observation)

실제 실행 경험상, 다음과 같은 현상이 반복적으로 관찰되었다.

* **S1 (Structure / Table generation)** 은 비교적 안정적으로 통과
* **S2 (Execution / Card generation)** 에서 FAIL이 더 자주 발생
* 직관적으로는 S1이 더 복잡해 보이나, 실제로는 S2가 더 불안정함

이 문서는 해당 현상이 **우연이나 구현 미숙이 아니라, 파이프라인 설계상 필연적인 결과**임을 설명한다.

---

## 2. 표면적 복잡도 vs 실제 실패 위험

| 구분       | S1                | S2                  |
| -------- | ----------------- | ------------------- |
| 겉보기 복잡도  | 높음 (테이블, 다필드, 구조) | 낮음 (카드 생성)          |
| 실제 실패 위험 | 낮음                | **높음**              |
| 실패 원인 성격 | 형식적 미흡            | **계약 위반, 카디널리티 오류** |

→ **겉보기 복잡도와 실패 위험은 반비례**한다.

---

## 3. S1이 상대적으로 안정적인 구조적 이유

### 3.1 S1은 “결정”이 아닌 “구조화” 단계

S1의 역할은:

* 개념 나열
* 관계 정리
* 필드 채우기

즉, 다음과 같은 제약이 없다.

* 정확한 개수(`len == N`)
* 출력 완결성 보장
* 강한 Fail-Fast 조건

결과적으로:

* 다소 길거나
* 표현이 모호해도
* **구조만 유지되면 통과 가능**

---

### 3.2 JSON / 테이블은 LLM의 강점 영역

LLM은 본질적으로:

* 반복 구조
* 키–값 채우기
* 포맷 모방

에 강하다.

따라서 S1은:

* 복잡해 보여도
* LLM의 자연스러운 생성 습성과 잘 맞아
  → 안정적으로 동작한다.

---

## 4. S2가 더 자주 실패하는 근본 원인

### 4.1 S2는 “조합적 제약” 단계

S2는 단순해 보이지만, 동시에 다음을 요구한다.

1. **정확히 N개 생성**
2. 각 카드마다:

   * front 비어 있지 않음
   * back 비어 있지 않음
3. 중복 금지
4. 허용된 카드 타입만 사용
5. 금지 필드 절대 불가

이는 단일 조건이 아니라 **여러 조건의 동시 만족**을 요구하는 단계다.

→ LLM 입장에서는 **난이도가 급격히 상승**한다.

---

### 4.2 S2는 LLM의 기본 생성 휴리스틱과 충돌

LLM의 자연스러운 행동:

* “중요한 것만 남기기”
* “중복 줄이기”
* “마지막에서 힘 빠짐”
* “적당히 요약”

그러나 S2의 요구는:

> “판단하지 말고, 최적화하지 말고, 정확히 N개를 끝까지 생성하라”

즉, **LLM의 본능을 억제하는 단계**다.

---

### 4.3 S2는 Fail-Fast가 처음으로 강하게 적용되는 지점

현재 파이프라인에서:

* S1: 구조만 맞으면 통과
* **S2: 처음으로 `len == N`, schema, forbidden field 등 하드 게이트 적용**

따라서:

> **S2는 설계 불일치가 최초로 ‘드러나는 지점’**

이다.

---

## 5. “S2에서 터진다”는 것의 의미

중요한 해석 포인트는 다음이다.

* 이것은 **퇴보**가 아니다.
* 코드 버그의 증거도 아니다.
* 오히려 **파이프라인이 정상적으로 작동하고 있다는 신호**다.

느슨한 시스템이라면:

* 잘못된 출력이
* 조용히 다음 단계로 흘러갔을 것이다.

현재 시스템은:

* 경계에서 정확히 멈춘다.
* 계약 위반을 즉시 드러낸다.

---

## 6. 운영적 결론

### 6.1 S1 안정화는 현재 우선순위 아님

* S1은 구조적으로 안정적
* 실패 빈도 낮음

### 6.2 S2는 “전용 안정화 프로젝트” 대상

향후 개선의 초점은 명확하다.

* S2 프롬프트 단순화
* JSON 스키마 기반 출력 강제
* 판단/권고/최적화 표현 완전 제거
* `len == N`을 프롬프트 + 코드에서 이중 강제

---

## 7. 한 문장 요약 (기록용)

> **S1은 복잡하지만 느슨한 단계이고,
> S2는 단순하지만 가장 엄격한 단계다.
> 따라서 S2에서 더 자주 실패하는 것은 설계상 정상이다.**
