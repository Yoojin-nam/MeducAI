# Group ID Migration Risks and Mitigation

**Status:** Analysis Document  
**Date:** 2025-12-21  
**Context:** Migration from `make_stable_group_id()` (16-char hash) to CSV `group_id` (`grp_` + 10-char hash)

---

## 1. Current Situation

### 1.1 Two Group ID Formats in Codebase

1. **Legacy format** (16-char hash): `d2da903bffbe2cd6`
   - Generated by `make_stable_group_id()` from components
   - Used in older S1 files (e.g., `S0_QA_fixed_v4`)

2. **Canonical format** (CSV): `grp_f073599bec`
   - Generated by `stable_group_id_from_key()` from `group_key`
   - Used in `groups_canonical.csv` (SSOT)
   - Now enforced in new S1 files

### 1.2 Code Changes Made

- ✅ `normalize_row()` now preserves CSV `group_id`
- ✅ `process_single_group()` uses CSV `group_id` (no fallback generation)
- ✅ S2-only mode checks both `group_id` and `group_key` for matching
- ✅ Preflight check no longer requires `group_id` in prompts

---

## 2. Identified Risks

### 2.1 **CRITICAL: S2-Only Mode Compatibility**

**Risk:** S2-only mode trying to match new CSV `group_id` (`grp_xxx`) against old S1 files with legacy `group_id` (16-char hash).

**Impact:** S2-only mode will fail to find S1 outputs for groups that were generated with old format.

**Current Mitigation:**
- S2-only mode checks both `group_id` AND `group_key` for matching
- This should work for most cases, but not guaranteed

**Example:**
```python
# Old S1 file has: group_id="d2da903bffbe2cd6"
# CSV has: group_id="grp_f073599bec"
# S2-only mode looks for: group_id="grp_f073599bec" → NOT FOUND
# Falls back to: group_key="bone_soft_tissue__diagnostic_imaging__tumor" → FOUND ✅
```

**Recommendation:**
- ✅ Already implemented: dual matching (group_id OR group_key)
- ⚠️ Monitor S2-only runs for any remaining mismatches

---

### 2.2 **HIGH: File Path and Filename Dependencies**

**Risk:** Many scripts use `group_id` in file paths and filenames. Format change affects:
- Image filenames: `IMG__{run_tag}__{group_id}__{entity_id}__{card_role}.jpg`
- PDF filenames: `SET_{group_id}_arm{arm}_{run_tag}.pdf`
- Allocation paths: `allocation_s0__group_{group_id}__arm_{arm}.json`

**Impact:** 
- New files will have `grp_xxx` format
- Old files have 16-char hash format
- Scripts expecting one format may not find files with the other

**Affected Components:**
1. **S4 Image Generator** (`04_s4_image_generator.py`)
   - Uses `group_id` in `make_image_filename()`
   - ✅ Should work (sanitizes any format)

2. **PDF Builder** (`07_build_set_pdf.py`)
   - Uses `group_id` in PDF filename: `SET_{group_id}_arm{arm}_{run_tag}.pdf`
   - ✅ Should work (just string formatting)

3. **Allocation** (`s0_allocation.py`)
   - Uses `group_id` in path: `allocation_s0__group_{group_id}__arm_{arm}.json`
   - ✅ Should work (just string formatting)

**Recommendation:**
- ✅ No code changes needed (all use string formatting, not parsing)
- ⚠️ Document that filenames will change format going forward

---

### 2.3 **MEDIUM: Cross-Run Compatibility**

**Risk:** Mixing old and new runs in same workflow.

**Scenario:**
- Old run: `S0_QA_fixed_v4` with 16-char `group_id`
- New run: `FULL_PIPELINE_V8_20251221_224604` with `grp_xxx` format
- Scripts that aggregate across runs may fail

**Affected Scripts:**
- `distribute_qa_packets.py`: Aggregates PDFs across runs
- `s0_noninferiority.py`: Compares results across runs
- Any analysis scripts that join on `group_id`

**Recommendation:**
- ⚠️ Add validation/warning if `group_id` format mismatch detected
- ⚠️ Consider migration script to normalize old S1 files (if needed)

---

### 2.4 **LOW: Database/Index Dependencies**

**Risk:** If any external systems (databases, search indices) use `group_id` as key.

**Impact:** Low (no evidence of external systems in codebase)

**Recommendation:**
- ✅ Monitor if any external integrations are added

---

### 2.5 **LOW: Human-Readable References**

**Risk:** `grp_xxx` format is more readable than 16-char hash, but still not human-friendly.

**Impact:** Minimal (group_id is not meant to be human-readable; `group_key` is)

**Recommendation:**
- ✅ No action needed (use `group_key` for human-facing operations)

---
