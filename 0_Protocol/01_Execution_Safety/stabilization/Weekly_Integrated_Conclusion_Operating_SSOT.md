**Primary goal:** Stabilize **Stage 1 (S1)** outputs so downstream stages (S2/S4/QA) do not collapse due to malformed or drifting inputs.  
**Operating principle:** **Fail-fast by Gate**, not fail-late downstream.

## 2) Problem Statement (Root Causes)

### 2.1 S1 output instability collapses S2+
- Master table (visual_type_category table) format drift: column titles, row structure, markdown breakage
- Entity list instability: too many / too few / mismatch with table rows
- Downstream receives **invalid structure**, then fails unpredictably

### 2.2 Objective input contamination
- Passing raw JSON-stringified objectives into prompts causes the model to parse **format** instead of **content**
- Copy/paste/mapping errors amplify into output drift

### 2.3 Large groups explode output
- 30–40 objectives trigger “enumeration mode”: too many rows, synonym inflation, token/cost spikes

### 2.4 “Temporary if/else workarounds” accumulate
- Local patches to pass one case create global regressions and schema drift
- File-replacement workflow increases blast radius of syntax mistakes

### 2.5 “Two Canonicals” (contract split) created validator/downstream conflict
- Competing definitions for `entity_list` type, mandatory fields, enum sets
- Gate and S2 interpret the same file differently

## 3) Decisions (Operating SSOT)

### D1) `objective_list` must be normalized to `objective_bullets` before S1
**Rule:** LLM must receive **only** `objective_bullets` (readable bullet list).  
**Fail-fast:** If JSON parse fails or objectives are empty, **do not call S1**.

**Normalization requirements:**
- JSON parse must succeed
- Trim whitespace
- Remove difficulty suffix only when present at sentence end: `(A)`, `(B)`, `(C)`
- Render as Markdown bullets (one objective per bullet)

---

### D2) Large-group summarization is a prompt+gate responsibility (not preprocessing)
**Rule:** Preprocessing only normalizes input.  
**Rule:** Row/Entity constraints and summarization are enforced by **S1 prompt + S1 Gate**.

---

### D3) `visual_type_category` enum (deprecated: Other removed)
**Rule:** `visual_type_category` enum is limited to 11 canonical categories (Other and additional categories deprecated).  
---

### D4) `entity_id` is **MUST** and **code-owned deterministic**
**Rule:** Every entity in final S1 struct has `entity_id`.  
**Rule:** `entity_id` is **NOT** generated by the LLM. It is injected by code after parsing/normalizing LLM output.

**Deterministic format (recommended):**
- `{group_id}__E{index:02d}` (stable order = table row order)
- Example: `G0123__E01`

**SSOT join keys:**
- **Group-level:** `group_id`
- **Entity-level:** `(group_id, entity_id)`

`entity_name` is a **display label**, not a join key.

---

### D6) S1 Master Table: hard format constraints (to prevent markdown/JSON corruption)
**Hard constraints to enforce in prompt + gate:**
- Exactly **one** master table
- Column headers are **EXACT match** to the canonical set used by the project
- Fixed column count (e.g., 6 columns if that is the canonical table)
- Each data row has exactly the same number of cells as headers
- No cell contains raw `|` characters (must be replaced/escaped by a defined rule)
- No multiline cells
- No empty cells

---

### D7) Row/Entity count limits must be consistent across prompt, gate, and docs
This week’s reports contained **a conflict** between:
- Hard cap **8–14 rows** (strict)
- Versus **Prefer 12–18, Hard max 20** (relaxed for broad groups)

**Operating SSOT resolution (effective immediately):**
- **Hard max:** 20 rows/entities  
- **Preferred range:** 10–14 rows/entities  
- **Large group (>14 objectives):** must still compress; may use **15–20** if necessary to avoid content collapse.

**Enforcement model:**
- `rows > 20` → **FAIL**
- `rows in 15–20` → **PASS with Warning** (log it)
- `rows in 10–14` → target

Rationale: strict 14-row cap can over-compress broad groups, causing drift/omissions and destabilizing outputs.

## 4) S1 Gate (Fail-fast) — Minimum PASS Contract

S1 Gate must reject anything that violates the **hard constraints** below.

### 4.1 Required artifacts
- `stage1_struct.jsonl` exists and is non-empty  
- Optional: `debug_raw` exists (for diagnosis), but not required for PASS if struct is valid

### 4.2 JSON/Schema invariants
- Valid JSON object per line (no trailing junk)
- Required keys exist and are correct types

### 4.3 Visual category rules
- `visual_type_category` is non-empty

### 4.4 Table integrity rules (hard)
- Exactly 1 table
- Header exact match, fixed column count
- All rows have same number of cells
- No `|` in cells, no multiline, no empty cells
- **Data rows <= 20**

### 4.5 Entity rules (hard)
- `entity_list` length equals the number of data rows
- Entities are objects containing `entity_id` and `entity_name` (minimum)
- `entity_id` exists, is unique, deterministic, stable order
- Entity names match the “item/concept” column text **exactly** (including order), if that rule is in force

## 5) S2 Contract (Interface Constraints)

### 5.1 Join key is `(group_id, entity_id)`
- S2 selection and execution must be entity_id–based
- `entity_name` selection is optional at most; if allowed, it must be a unique match or rejected

### 5.2 S2 consumes only **validated fields**
- S2 may only rely on S1 fields that are gate-validated
- Any additional “nice-to-have” fields must not block S2 execution unless explicitly promoted to “hard” by SSOT update

---

## 6) Operational Reliability Rules (Non-negotiable)

### OR1) No “temporary if/else workaround” without contract + tests
- If an input class breaks, fix the **contract/prompt/gate** or the root transformation.
- Do not add silent fallbacks that produce artifacts that “look OK” but violate schema.

### OR2) File replacement workflow must include compile+smoke+gate before commit
Minimum sequence:
1) Replace file
2) `python -m compileall 3_Code/src` (or equivalent)
3) Smoke run with a short group (`--row_index` or `--only_group_id`)
4) Run S1 Gate
5) Only then commit/push

### OR3) Retry/backoff for transient model errors
- Use bounded retries (e.g., 3–5 attempts) with exponential backoff + jitter
- Log: exception type, attempt count, final status

## 7) Acceptance Criteria (Definition of “S1 Stabilized”)

S1 is considered “stabilized enough to proceed” when:

  - `stage1_struct.jsonl` always generated
  - S1 Gate PASS (hard rules)
  - Row/entity limits obeyed (<=20)
  - `entity_id` deterministic and preserved
  - No schema ambiguity (validator/docs aligned)

## 12) S0 Non-Inferiority Analysis (2025-12-20)

**Effective Date:** 2025-12-20

### 12.1 Canonical Reference

S0 non-inferiority analysis for arm selection is governed by:
- **Canonical criteria:** `0_Protocol/06_QA_and_Study/QA_Operations/S0_Noninferiority_Criteria_Canonical.md`
- **Analysis script:** `3_Code/src/tools/qa/s0_noninferiority.py`

### 12.2 Key Specifications

- **Two-layer decision framework:**
  1. Safety endpoint: prevents unacceptable increases in major errors (0-score cards)
  2. Primary NI endpoint: mean accuracy score non-inferiority test
- **Non-inferiority margin (Δ):** 0.05 (default), configurable to 0.03 (conservative) or 0.08 (cost-driven)
- **CI level:** 90% (default) for operational selection
- **Statistical method:** Clustered paired bootstrap over (rater_id, group_id) pairs

### 12.3 Usage

```bash
python 3_Code/src/tools/qa/s0_noninferiority.py \
  --input_csv <path> \
  --baseline_arm A \
  --delta 0.05 \
  --ci 0.90 \
  --bootstrap_n 5000 \
  --seed 20251220 \
  --out_dir <dir>
```

Outputs are written to `{out_dir}/qa_s0_noninferiority_*.{csv,md,json}` with deterministic naming.
