# File Replacement Patch Delivery Rule

Status: Canonical  
Version: 1.1  
Applies to: All Steps (S0–FINAL)  
Location: `0_Protocol/01_Execution_Safety/`  
Last Updated: 2025-12-18  

## 0) 핵심 목표

이 규칙은 **diff를 읽지 않는(또는 최소화하는) 운영 방식**을 유지하면서도, 다음 실패를 구조적으로 줄이기 위한 것이다.

- “임시 우회(if/else, 하드코딩)”가 누적되어 **끝까지 갔다가 폭발**하는 붕괴
- 파일 통째 교체 과정에서 발생하는 **의도치 않은 변경**과 **문법 오류**
- 실패 원인을 덮어쓴 채 다음 단계로 진행하는 **재현성 붕괴**

---

## 1) 비협상 원칙 (Non‑Negotiable)

1. **diff(줄 단위 수정) 금지**: 기본 전달 단위는 “파일 1개 = 완성본 1개”다.  
2. 변경은 항상 아래 중 하나로 표현한다.  
   - **REPLACE**: 기존 파일을 통째로 교체  
   - **ADD**: 새 파일 추가  
   - **DELETE**: 파일 삭제(가능하면 최소화)  
3. 패치는 항상 **“패치 단위(PATCH_ID)”**로 묶어서 제공한다. 한 번에 여러 건을 섞지 않는다.  
4. 패치 적용 후에는 반드시 **터미널 기반 검증 커맨드**를 함께 제공한다.  
5. 실패 시 즉시 되돌릴 수 있도록 **rollback 명령**을 항상 포함한다.  

### 1.1 Anti‑Bypass Rule (이번 업데이트의 핵심)

> **임시 우회(if/else로 예외 처리, 고정 문자열/하드코딩, try/except로 삼키기)로 “끝까지 가기”를 절대 허용하지 않는다.**  
> **원인은 반드시 제거하며, 변수가 정상 경로로 흘러가게 한다.**

- “일단 돌아가게”가 아니라 **“정석 경로를 복구”**가 목표다.
- 우회가 필요해 보이는 순간은, 거의 항상 **입력/계약/경로/스키마/인바리언트의 불일치**를 의미한다.
- Fail‑Fast 정책과 충돌하는 “완화/보정/계속 진행”은 **금지**다.

## 2) 내가 앞으로 너에게 주는 “패치 메시지”의 고정 구조

### (A) 헤더

- **PATCH_ID**: `PATCH_YYYYMMDD_HHMM_<slug>`
- **목적(1~2줄)**: 무엇을 고치고 어떤 실패를 막는지
- **영향 범위**: Step/모듈/출력경로(RUN_TAG 등)
- **리스크 레벨**: Low / Medium / High
- **호환성**: (예) “S0 모드/DEV_SMOKE 모두 OK”, “allocation 미적용 전제” 등
- **검증 기준(필수)**: “성공이면 무엇이 PASS인지”를 1~3줄로 선언

### (B) 적용 전 준비(당신이 실행)

```bash
git add -A
git commit -m "snapshot: before PATCH_ID"
git status
```

### (C) 변경 목록(반드시 3분류로만)

- **REPLACE**
  - `상대경로/파일명` — 왜 교체하는지(1줄)
- **ADD**
  - `상대경로/파일명` — 왜 추가하는지(1줄)
- **DELETE**
  - `상대경로/파일명` — 왜 삭제하는지(1줄)

### (D) “파일 본문” 제공 규칙(가장 중요)

각 파일은 아래 템플릿으로 제공한다.

```
=== FILE: REPLACE path/to/file.py ===
<파일 전체 내용>
=== END FILE ===
```

(ADD도 동일, DELETE는 본문 없이 명령만)

## 3) “파일 교체형”을 안전하게 만드는 추가 가드레일

### 3.1 변경 범위 제한 (Damage Containment)

- **한 PATCH에서 REPLACE는 1–2개 파일이 원칙**  
  예외적으로 3개 이상이 필요하면:
  - “왜 분리 불가능한지”
  - “각 파일의 변경 이유”
  - “검증 커맨드가 무엇을 커버하는지”
  를 명시한다.

### 3.2 불변 구간 봉인 (Editable Region)

파일마다 수정 가능/금지 영역을 명시한다.

```python
# ==== EDITABLE REGION START (ALLOWED) ====
# 이 블록 안만 수정 가능
# ==== EDITABLE REGION END (ALLOWED) ====

# ==== DO NOT EDIT BELOW (INVARIANT) ====
# 계약/스키마/핵심 함수 시그니처/경로 규칙
```

운영 규칙:

- PATCH는 **editable region 안에서만 변경**이 기본이다.
- 불가피하게 밖을 바꾸면, “왜/어떤 위험/어떤 테스트로 방어했는지”를 패치 헤더에 포함한다.

### 3.3 커밋 전 최소 자동검증(필수 3종)

**(1) 문법 오류 차단**

```bash
python -m compileall 3_Code/src
```

**(2) 빠른 스모크 실행(샘플 1)**  
프로젝트가 보유한 `Scripts/smoke_*.sh` 또는 `--sample 1` 기반 경량 실행을 사용한다.

**(3) 필수 산출물 존재 확인(Manifest/경로)**  
RUN_TAG 하위 필수 아티팩트(예: step01 jsonl, allocation artifact)가 생성되지 않으면 커밋하지 않는다.

## 4) 개발자 관점 “정석” 디버깅/개발 원칙 (초보 친화 버전)

아래는 “개발자라면 기본으로 지키는” 습관을 **파일 교체형 워크플로**에 맞춰 번역한 것이다.

### 4.1 Reproduce → Isolate → Fix → Prove (4‑Step Loop)

1. **재현(Reproduce)**: 동일 RUN_TAG/동일 입력으로 **같은 에러가 다시 나야** 한다.  
2. **격리(Isolate)**: 에러가 발생하는 **최소 범위(파일/함수/입력 1개)**를 만든다.  
   - 가능하면 `--sample 1`, 단일 group, 단일 arm으로 좁힌다.
3. **원인 제거(Fix Root Cause)**:  
   - 우회(if/else)로 “에러를 안 보이게” 만드는 대신  
   - **변수/계약/경로/스키마**를 정상 상태로 되돌린다.
4. **증명(Prove)**: 최소 검증 3종(compileall + smoke + artifact)을 통과시키고 커밋한다.

### 4.2 “고정 문자열이 있는데도 되는 줄 알았다” 유형의 예방책

- **계약 위반은 즉시 FAIL**: 빈 값/누락 값을 “기본값”으로 채워 넣지 않는다.  
- “일단 진행” 대신, **문제 지점에서 멈추고(FAIL) 로그로 남긴다.**
- 값이 고정이어야 한다면:
  - 상수로 선언(한 곳에서만)
  - 검증 assert를 둬서 변경 시 즉시 실패

### 4.3 버전 기록(실수 방지) — 작은 커밋을 쌓는다

권장 리듬:

- (A) 작은 변경 → (B) compileall → (C) smoke → (D) 커밋  
이 사이클을 **자주** 반복한다.  
“통째로 크게 바꾸고 마지막에 한 번 실행”은 실패 확률이 급격히 커진다.

### 4.4 로그/에러 메시지의 원칙

- 에러는 “친절하게”:
  - 어떤 값이 기대되었는지
  - 실제 값이 무엇이었는지
  - 어떤 계약/정책을 위반했는지
- 예외를 잡아도 `pass`로 삼키지 않는다.  
  잡았다면 **즉시 FAIL + 원인 메시지 기록**이 원칙이다.

---

## 5) 운영 상 “정상/비정상” 판정 규칙

- **PASS**: 최소 검증 3종을 통과하고, 필수 아티팩트가 생성됨
- **FAIL**: 어느 하나라도 실패하면, 다음 단계 진행/배포/QA 평가에 사용 금지  
- FAIL 결과는 “버리지 않는다”: 로그와 산출물은 **감사/재현 목적**으로 남긴다(배포 사용은 금지)

---

## 6) 실패 시 복구(항상 포함)

```bash
git reset --hard HEAD
```

또는 “패치 적용 전 커밋으로 복귀”를 원하면:

```bash
git reset --hard HEAD~1
```
