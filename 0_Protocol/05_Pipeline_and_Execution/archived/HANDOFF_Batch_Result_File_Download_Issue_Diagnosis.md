# 배치 결과 파일 다운로드 병렬 처리 문제 진단 및 개선 방안

**작성일**: 2026-01-15  
**상태**: 문제 진단 중 (개선 필요)

---

## 📋 현재 문제 상황

### 증상
- `--check_status` 모드에서 결과 파일 다운로드를 병렬로 처리하도록 개선했으나, **다운로드가 멈춤**
- "📥 Downloading result file..." 메시지는 출력되지만, "✅ Result file downloaded successfully" 메시지가 나오지 않음
- 타임아웃을 추가했지만, 근본 원인 해결이 아님

### 이전 동작 (정상 작동)
- **단일 API 키 + 단일 그룹 테스트**: 정상 작동 ✅
- 순차 처리: 각 배치 결과 파일을 하나씩 다운로드 → 느리지만 정상 작동

### 현재 동작 (문제 발생)
- **병렬 처리 (워커 5-10개)**: 다운로드가 멈춤 ❌
- 여러 배치의 결과 파일을 동시에 다운로드 시도 → 일부 또는 전체가 멈춤

---

## 🔍 원인 분석

### 가능한 원인들

#### 1. API Rate Limit / Quota 제한 (가능성 높음)
- **증상**: 여러 키를 동시에 사용할 때 제한에 걸릴 수 있음
- **배경**: 
  - 각 배치는 서로 다른 API 키로 생성됨
  - 병렬로 여러 키에서 동시 다운로드 시도
  - Google API의 동시 연결/요청 제한에 걸릴 수 있음
- **확인 방법**: 단일 워커로 테스트

#### 2. 네트워크 대역폭 제한
- **증상**: 동시에 너무 많은 다운로드 요청
- **배경**: 
  - 결과 파일이 상대적으로 큰 JSONL 파일 (수백 KB ~ 수 MB)
  - 5-10개를 동시에 다운로드하면 네트워크 병목 발생 가능
- **확인 방법**: 워커 수를 1-2개로 줄여서 테스트

#### 3. API 서버 측 동시 연결 제한
- **증상**: 서버가 동시 연결 수를 제한할 수 있음
- **배경**: 
  - `client.files.download()`는 HTTP 요청
  - 서버 측에서 IP/세션당 동시 연결 제한 가능
- **확인 방법**: 단일 워커로 테스트

#### 4. 파일 크기 / 다운로드 시간
- **증상**: 일부 파일이 커서 다운로드 시간이 오래 걸림
- **배경**: 
  - 배치 결과 파일 크기는 배치 크기에 비례 (수백 개 이미지 = 수 MB)
  - 네트워크가 느리면 개별 파일 다운로드에도 시간 소요
- **확인 방법**: 로그에서 실제 다운로드 시간 측정

#### 5. API 키 인증 충돌 (가능성 낮음)
- **증상**: 여러 스레드에서 같은 키를 동시에 사용
- **배경**: 
  - 각 배치는 생성 시 사용한 키로 다운로드해야 함
  - 스레드 간 키 접근 충돌 가능성 (하지만 코드상으로는 각 스레드가 독립적)
- **확인 방법**: 코드 리뷰

#### 6. `client.files.download()` 내부 구현 문제
- **증상**: Python SDK의 `files.download()`가 동시 호출 시 문제 발생
- **배경**: 
  - SDK 내부적으로 연결 풀링이나 세션 관리 문제 가능
  - ThreadPoolExecutor와의 상호작용 문제
- **확인 방법**: 순차 처리로 되돌려서 테스트

## 🎯 개선 방안

### 방안 1: 단계별 접근 - 상태 확인만 병렬, 다운로드는 순차 (권장)

**접근 방법**:
1. **상태 확인 단계**: 순차 처리 유지 (이미 빠름)
2. **결과 파일 다운로드**: 순차 처리로 되돌림
3. **이미지 다운로드**: 병렬 처리 유지 (기존 로직)

**장점**:
- 검증된 동작 방식으로 되돌림
- 추가 문제 발생 가능성 낮음
- 상태 확인은 빠르므로 병렬 처리 불필요

**단점**:
- 결과 파일 다운로드는 느림 (기존 문제 재발)

**구현**:
- `download_result_files_parallel()` 함수 제거 또는 사용 안 함
- 결과 파일 다운로드를 순차 처리로 되돌림

---

### 방안 2: 제한적 병렬 처리 - 워커 수 1-2개로 제한

**접근 방법**:
- 병렬 처리는 유지하되, 워커 수를 매우 낮게 설정 (1-2개)
- API 서버 부하를 최소화하면서 일부 성능 향상

**장점**:
- 병렬 처리의 이점을 일부 활용
- API 제한에 걸릴 가능성 낮음

**단점**:
- 성능 향상이 제한적
- 근본 원인 해결이 아님

**구현**:
- `--max-download-workers` 기본값을 1 또는 2로 변경
- 또는 결과 파일 다운로드 전용 플래그 추가

---

### 방안 3: 재시도 로직 + 타임아웃 강화

**접근 방법**:
- 타임아웃을 더 짧게 설정 (30초)
- 타임아웃 발생 시 재시도 (최대 2-3회)
- 재시도 간격 추가 (exponential backoff)

**장점**:
- 일시적인 네트워크 문제 해결 가능
- API 서버 일시 부하 해결 가능

**단점**:
- 근본 원인 해결이 아님
- 여전히 멈출 수 있음

**구현**:
- `download_result_file_for_batch()` 함수에 재시도 로직 추가
- 타임아웃을 30초로 단축

---

### 방안 4: 하이브리드 접근 - 작은 배치 단위 병렬 처리

**접근 방법**:
- 모든 파일을 한 번에 병렬 처리하지 않고
- 작은 배치(예: 3-5개)로 나누어 순차 처리
- 각 배치 내에서는 병렬 처리

**장점**:
- 병렬 처리의 이점 활용
- API 부하 분산

**단점**:
- 구현 복잡도 증가
- 성능 향상이 제한적

---

## 🔧 즉시 시도할 개선안 (권장)

### 단기 해결책: 상태 확인만 병렬, 다운로드는 순차

**이유**:
1. 상태 확인은 빠르므로 병렬 처리 필요성 낮음
2. 결과 파일 다운로드만 순차로 처리해도 전체 시간에 큰 영향 없음
3. 검증된 동작 방식으로 안정성 확보

**구현 단계**:
1. `download_result_files_parallel()` 함수 호출 제거
2. 결과 파일 다운로드를 순차 처리로 되돌림
3. 테스트 및 검증
