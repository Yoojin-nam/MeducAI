## 📋 현재 문제 상태

### 문제
`--check_status` 모드에서 배치 상태 확인 시, **결과 파일 다운로드가 순차적으로 실행**되어 시간이 오래 걸립니다.

### 현재 동작 (라인 2426-2477)
1. 각 배치의 상태를 순차적으로 확인
2. SUCCEEDED 상태인 배치에 대해:
   - `prompts_metadata`만으로 이미지 존재 여부를 먼저 확인
   - 모든 이미지가 없으면 결과 파일(`download_batch_results`)을 **순차적으로** 다운로드 (라인 2445)
   - 결과 파일을 파싱하여 정확한 이미지 존재 여부 확인
3. 다운로드가 필요한 배치들을 큐에 추가
4. 모든 배치 확인이 끝난 후 병렬로 이미지 다운로드

### 병목 지점
- **라인 2445**: `result_file_bytes = download_batch_results(dest_file_name, api_key=download_key)`
- 각 배치마다 순차적으로 API 호출하여 결과 파일 다운로드
- 약 30개 배치 × 각 배치당 몇 초 = 수 분 소요

### 목표

### 구현 방안

#### 접근 방법 1: 2단계 병렬 처리 (권장)
1. **1단계**: 모든 SUCCEEDED 배치를 수집 (순차, 빠름)
   - 상태 확인만 수행 (이미 순차 실행 중)
   - 이미지 존재 여부를 `prompts_metadata`로 빠르게 체크
   - 결과 파일 다운로드가 필요한 배치만 별도 리스트에 추가

2. **2단계**: 결과 파일 다운로드를 병렬로 처리
   - `ThreadPoolExecutor`를 사용하여 결과 파일 다운로드
   - 각 배치의 `batch_results`를 저장
   - 결과 파일로 정확한 이미지 존재 여부 확인

3. **3단계**: 이미지 다운로드 (기존 로직 재사용)
   - `download_batches_parallel()` 함수 사용
