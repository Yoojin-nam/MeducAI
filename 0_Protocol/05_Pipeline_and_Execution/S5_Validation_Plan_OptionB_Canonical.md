## 0. Executive Summary

This plan implements S5 Validation as an LLM-based triage/flagging step that provides evidence to human raters **without contaminating primary endpoints**. The workflow uses a **2-pass rating system**:

1. **Pre-S5 Pass**: Human rates content **before** seeing S5 results (primary endpoint for arm comparison)
2. **S5 Reveal**: S5 validation results are revealed after Pre-S5 submission is locked
3. **Post-S5 Pass**: Human may correct ratings with mandatory change logging (secondary/exploratory endpoint)

**Critical Design Principle**: Pre-S5 ratings are **immutable** and used for all arm comparison analyses. Post-S5 ratings measure tool effect, not arm performance.

## 1. Endpoint & Estimand Definitions

### 1.1 Primary Endpoints (Pre-S5 Ratings Only)

**For Arm Comparison (Non-Inferiority Analysis):**

#### 1.1.1 Blocking Error Rate (BER_pre)
- **Definition**: Proportion of cards with `blocking_error_pre = true` per arm
- **Unit**: Card-level (binary: blocking error present/absent)
- **Aggregation**: Arm-level mean proportion across all evaluated cards
- **Notation**: `BER_pre_arm = mean(blocking_error_pre) for arm`
- **Use**: Primary safety endpoint for arm comparison
- **Analysis**: Non-inferiority test (candidate vs baseline) with Δ margin

#### 1.1.2 Overall Card Quality (GCR_pre)
- **Definition**: Mean Overall Card Quality score (1-5 Likert) per arm
- **Unit**: Card-level (1-5 Likert scale)
- **Aggregation**: Arm-level mean across all evaluated cards
- **Notation**: `GCR_pre_arm = mean(overall_quality_pre) for arm`
- **Use**: Primary quality endpoint for arm comparison (optional, if collected)
- **Analysis**: Non-inferiority test (candidate vs baseline) with Δ margin

**Decision Rule**: All arm comparison decisions (selection, non-inferiority) use **Pre-S5 ratings only**.

### 1.2 Secondary/Exploratory Endpoints (Post-S5 Ratings)

**For Tool Effect Measurement:**

#### 1.2.1 Blocking Error Correction Rate (BER_delta)
- **Definition**: `BER_post - BER_pre` (change in blocking error rate after S5 reveal)
- **Notation**: `BER_delta = BER_post - BER_pre`
- **Interpretation**: Positive delta = S5 helped catch more errors; negative delta = S5 caused false positives
- **Use**: Measure S5 tool effectiveness (exploratory)

#### 1.2.2 Quality Score Change (GCR_delta)
- **Definition**: `GCR_post - GCR_pre` (change in mean quality score after S5 reveal)
- **Notation**: `GCR_delta = GCR_post - GCR_pre`
- **Interpretation**: Positive delta = S5 improved quality assessment; negative delta = S5 degraded assessment
- **Use**: Measure S5 tool effect on quality perception (exploratory)

#### 1.2.3 Correction Time (CT_post)
- **Definition**: Time spent in Post-S5 pass (seconds, from S5 reveal to final submission)
- **Unit**: Card-level (continuous, seconds)
- **Aggregation**: Mean correction time per card, per arm
- **Use**: Measure S5 tool efficiency (exploratory)

### 1.3 S5 Tool Performance Endpoints (Exploratory)

#### 1.3.1 S5 Precision at Blocking
- **Definition**: Among cards where S5 flagged blocking error, proportion where human also flagged (Pre or Post)
- **Notation**: `Precision@Blocking = P(human_blocking | S5_blocking)`
- **Use**: Measure S5 false positive rate

#### 1.3.2 S5 Recall at Blocking (Audit-based)
- **Definition**: Among cards where human flagged blocking error (Pre or Post), proportion where S5 also flagged
- **Notation**: `Recall@Blocking = P(S5_blocking | human_blocking)`
- **Use**: Measure S5 false negative rate (requires audit if feasible)

**Note**: Recall measurement may require additional audit sampling if Post-S5 corrections are sparse.

### 1.4 Estimand Clarity Statement

**Primary Estimand (Arm Comparison):**
- **Population**: All cards generated by candidate arms (A, B, C, D) vs baseline (E)
- **Intervention**: Arm configuration (model, thinking, RAG settings)
- **Comparison**: Candidate arm vs baseline arm
- **Outcome**: `BER_pre` and/or `GCR_pre` (Pre-S5 ratings only)
- **Analysis**: Non-inferiority test with predefined Δ margin

**Secondary Estimand (Tool Effect):**
- **Population**: All cards evaluated in FINAL QA
- **Intervention**: S5 validation tool (revealed after Pre-S5 lock)
- **Comparison**: Post-S5 vs Pre-S5 ratings
- **Outcome**: `BER_delta`, `GCR_delta`, `CT_post`
- **Analysis**: Descriptive statistics, correlation with S5 flags

**Critical Constraint**: Pre-S5 ratings are **never** influenced by S5 results, ensuring clean arm comparison.

## 2. Two-Pass (Pre → Reveal S5 → Post) Evaluation Workflow

### 2.1 Workflow States

```
[Initial] → [Pre-S5 Rating] → [Pre-S5 Locked] → [S5 Revealed] → [Post-S5 Rating] → [Final Submission]
```

### 2.2 Pre-S5 Pass (Primary Endpoint Collection)

#### 2.2.1 State: Pre-S5 Rating
- **UI State**: S5 results are **hidden** (not visible to rater)
- **Required Fields**:
  - `blocking_error_pre` (Yes/No, required)
  - `overall_quality_pre` (1-5 Likert, required)
  - `evidence_comment_pre` (conditional: required if `blocking_error_pre=Yes` OR `overall_quality_pre<=2`)
- **Optional Fields**:
  - `technical_accuracy_pre` (0/0.5/1, optional for FINAL)
  - `time_pre_ms` (auto-logged: time from card view to Pre-S5 submission)

#### 2.2.2 State: Pre-S5 Locked
- **Trigger**: User clicks "Submit Pre-S5 Rating" button
- **Action**: 
  - Pre-S5 fields become **read-only** (immutable)
  - Pre-S5 data is **saved to database** with timestamp
  - S5 reveal button becomes **enabled**
- **Locking Rules**:
  - Pre-S5 ratings **cannot be edited** after lock
  - Pre-S5 ratings are **final** for primary endpoint analysis
  - Lock timestamp is recorded: `pre_lock_timestamp_ms`

#### 2.2.3 UI Behavior
- **S5 Results Panel**: Hidden/collapsed, with visual indicator "S5 results will be revealed after Pre-S5 submission"
- **Submit Button**: "Submit Pre-S5 Rating" (primary action)
- **Validation**: All required fields must be filled before submission

### 2.3 S5 Reveal Gate

#### 2.3.1 Reveal Trigger
- **Condition**: Pre-S5 rating must be **locked** (submitted)
- **Action**: "Reveal S5 Results" button becomes enabled
- **User Action**: User clicks "Reveal S5 Results" button

#### 2.3.2 S5 Results Display
- **Content**: 
  - S5 blocking error flag (if any)
  - S5 quality assessment (if any)
  - S5 evidence/suggestions (structured display)
  - RAG source citations (if blocking error claimed)
- **Visual Design**:
  - Distinct panel/section (clearly separated from Pre-S5 ratings)
  - Visual indicator: "S5 Validation Results (for reference only)"
  - Color coding: Red for blocking errors, Yellow for warnings
- **Timestamp**: `s5_reveal_timestamp_ms` (logged when S5 is revealed)

#### 2.3.3 S5 Snapshot Versioning
- **S5 Snapshot ID**: `s5_snapshot_id` (hash of S5 validation result + S5 model version)
- **Purpose**: Ensure reproducibility; link human rating to exact S5 version
- **Format**: `s5_{run_tag}_{group_id}_{arm}_{s5_model_version}_{hash}`

### 2.4 Post-S5 Pass (Correction with Change Logging)

#### 2.4.1 State: Post-S5 Rating
- **UI State**: 
  - Pre-S5 ratings displayed as **read-only** (grayed out, with "Pre-S5" label)
  - Post-S5 fields displayed as **editable** (with "Post-S5" label)
  - S5 results visible (reference panel)
- **Default Values**: Post-S5 fields **pre-populated** with Pre-S5 values
- **User Action**: User may edit Post-S5 fields

#### 2.4.2 Change Detection & Logging
- **Change Detection**: Compare Post-S5 values to Pre-S5 values
- **Change Logging Rules**:
  - If **any** field changes, **mandatory** change log entry required:
    - `change_reason_code` (enum, required)
    - `change_note` (string, 1-line, required)
    - `change_timestamp_ms` (auto-logged)
  - If **no** fields change: No change log entry (Post-S5 = Pre-S5)

#### 2.4.3 Change Reason Codes (Enum)

```python
CHANGE_REASON_CODES = {
    "S5_BLOCKING_FLAG": "S5 flagged blocking error; confirmed after review",
    "S5_BLOCKING_FALSE_POS": "S5 flagged blocking error; determined to be false positive",
    "S5_QUALITY_INSIGHT": "S5 quality assessment provided useful insight",
    "S5_EVIDENCE_HELPED": "S5 evidence/suggestion helped identify issue",
    "S5_NO_EFFECT": "S5 results reviewed but no change needed",
    "RATER_REVISION": "Rater reconsidered without S5 influence (rare)",
    "OTHER": "Other reason (specify in note)"
}
```

**Special Case**: If `blocking_error` changes from `No→Yes` or `Yes→No`, `change_reason_code` is **mandatory** and must include explanation.

#### 2.4.4 Final Submission
- **Trigger**: User clicks "Submit Final Rating" button
- **Action**:
  - Post-S5 ratings saved to database
  - Change log saved (if any changes)
  - Correction time calculated: `correction_time_ms = final_submit_timestamp_ms - s5_reveal_timestamp_ms`
  - Session marked as complete

### 2.5 Workflow Guardrails

#### 2.5.1 Immutability Enforcement
- **Pre-S5 Lock**: Database-level constraint prevents Pre-S5 field updates after lock
- **UI Enforcement**: Pre-S5 fields disabled (read-only) after lock
- **Audit Trail**: All Pre-S5 submissions logged with timestamp and rater ID

#### 2.5.2 S5 Reveal Gating
- **Backend Check**: S5 reveal API endpoint verifies Pre-S5 lock exists before returning S5 results
- **Frontend Check**: S5 reveal button disabled until Pre-S5 submitted
- **Audit Trail**: S5 reveal timestamp logged

#### 2.5.3 Change Log Validation
- **Required Fields**: If Post-S5 differs from Pre-S5, `change_reason_code` and `change_note` are **required**
- **Validation**: Backend rejects final submission if changes exist without change log
- **Audit Trail**: All change log entries timestamped and linked to rater ID

## 4. UI/QA Form Changes (FINAL Form)

### 4.1 One-Screen Layout Concept

**Design Principle**: Minimal friction, clear separation of Pre-S5 and Post-S5 phases.

**Layout Structure**:
```
┌─────────────────────────────────────────────────┐
│ Card Content (Top)                              │
│ - Front / Back text                              │
│ - Card metadata (card_id, entity_name)          │
└─────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────┐
│ Pre-S5 Rating Section (Initially Active)        │
│ - B1: Blocking Error (Yes/No)                   │
│ - B2: Overall Quality (1-5)                     │
│ - B3: Evidence Comment (conditional)             │
│ [Submit Pre-S5 Rating] button                   │
└─────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────┐
│ S5 Results Panel (Hidden until reveal)           │
│ - S5 Blocking Error Flag (if any)               │
│ - S5 Quality Assessment                          │
│ - S5 Evidence/Suggestions                       │
│ [Reveal S5 Results] button (disabled until     │
│  Pre-S5 submitted)                              │
└─────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────┐
│ Post-S5 Rating Section (Hidden until S5 reveal) │
│ - Pre-S5 values (read-only, grayed)            │

│ - Change reason (required if changed)           │
│ [Submit Final Rating] button                    │
└─────────────────────────────────────────────────┘
```

### 4.2 Pre-S5 Rating Fields (Exact Specification)

#### B1. Blocking Error (Required)
- **Type**: Radio button (Yes/No)
- **Label**: "Does this card contain a blocking error?"
- **Tooltip**: "Blocking error = factual error that could mislead learners or cause incorrect clinical decisions"
- **Required**: Yes
- **Default**: None (must select)

#### B2. Overall Card Quality (Required)
- **Type**: Likert scale (1-5)
- **Label**: "Overall Card Quality"
- **Options**: 
  - 1 = 매우 나쁨 (Very Poor)
  - 2 = 나쁨 (Poor)
  - 3 = 보통 (Average)
  - 4 = 좋음 (Good)
  - 5 = 매우 좋음 (Very Good)
- **Required**: Yes
- **Default**: None (must select)

#### B3. Evidence Comment (Conditional)
- **Type**: Text area (1-line, max 200 chars)
- **Label**: "Evidence/Comment"
- **Condition**: Required if `B1=Yes` OR `B2<=2`
- **Placeholder**: "Brief explanation of issue or evidence"
- **Validation**: Must be non-empty if condition met

#### Submit Pre-S5 Rating Button
- **Label**: "Submit Pre-S5 Rating"
- **State**: Enabled when B1 and B2 are filled (and B3 if required)
- **Action**: 
  - Validate all required fields
  - Lock Pre-S5 fields (make read-only)
  - Save Pre-S5 data to backend
  - Enable "Reveal S5 Results" button
  - Show confirmation: "Pre-S5 rating submitted. You may now reveal S5 results."

### 4.3 S5 Results Panel (Reveal Logic)

#### Initial State (Hidden)
- **Visibility**: Hidden/collapsed
- **Indicator**: "S5 results will be revealed after Pre-S5 submission"
- **Button**: "Reveal S5 Results" (disabled, grayed out)

#### After Pre-S5 Submission
- **Button State**: "Reveal S5 Results" (enabled)
- **Action on Click**:
  - Fetch S5 validation result from backend (using `card_id`)
  - Display S5 results panel
  - Log `s5_reveal_timestamp_ms`
  - Show Post-S5 rating section

#### S5 Results Display
- **S5 Blocking Error Flag**:
  - If S5 flagged blocking: Red badge "S5: Blocking Error Detected"
  - If S5 did not flag: Green badge "S5: No Blocking Error"
- **S5 Quality Assessment**:
  - Display: "S5 Quality: {score}/5" (if available)
- **S5 Evidence/Suggestions**:
  - Display S5 issues list (if any)
  - Display RAG evidence citations (if blocking error claimed)
- **Visual Design**:
  - Distinct panel with border/background color
  - Header: "S5 Validation Results (for reference only)"
  - Footer: "These results are for your reference. Your Pre-S5 rating is locked and will be used for analysis."

### 4.4 Post-S5 Rating Fields (Exact Specification)

#### Pre-S5 Values Display (Read-Only)
- **Layout**: Grayed out, with "Pre-S5" label
- **Fields**: 
  - `blocking_error_pre` (read-only)
  - `overall_quality_pre` (read-only)
  - `evidence_comment_pre` (read-only, if exists)

#### Post-S5 Fields (Editable)
- **Default Values**: Pre-filled with Pre-S5 values
- **Fields**:
  - `blocking_error_post` (Yes/No, editable)
  - `overall_quality_post` (1-5 Likert, editable)
  - `evidence_comment_post` (text area, editable, conditional)

#### Change Reason (Required if Changed)
- **Trigger**: Any Post-S5 field differs from Pre-S5 value
- **Fields**:
  - `change_reason_code` (dropdown, required if changed)
  - `change_note` (text area, 1-line, max 200 chars, required if changed)
- **Validation**: 
  - If any field changed, both `change_reason_code` and `change_note` are required
  - Backend rejects submission if changes exist without change log

#### Submit Final Rating Button
- **Label**: "Submit Final Rating"
- **State**: Enabled when all required fields are filled
- **Action**:
  - Validate all required fields (including change log if changed)
  - Calculate `correction_time_ms`
  - Save Post-S5 data and change log to backend
  - Mark session as complete
  - Show confirmation: "Final rating submitted. Thank you!"

### 4.5 Conditional Logic Summary

**Pre-S5 Phase**:
- B3 (Evidence Comment) required if: `B1=Yes` OR `B2<=2`
- Submit button enabled if: B1 filled AND B2 filled AND (B3 filled if required)

**S5 Reveal Phase**:
- Reveal button enabled if: Pre-S5 submitted (locked)

**Post-S5 Phase**:
- Change reason required if: Any Post-S5 field differs from Pre-S5
- Submit button enabled if: All Post-S5 fields filled AND (change log filled if changed)

### 4.6 Time Measurement

**Pre-S5 Time**:
- Auto-logged: `time_pre_ms = timestamp_submitted_ms - timestamp_viewed_ms`
- No user input required

**Correction Time**:
- Auto-logged: `correction_time_ms = timestamp_final_submitted_ms - s5_reveal_timestamp_ms`
- No user input required (prefer logs)
- Fallback: If logs unavailable, use time bins: `<30s / 30s-1m / 1m-3m / 3m-5m / >5m`

## 5. Analysis Plan Implications

### 5.1 Primary Analysis (Arm Comparison)

**Endpoint**: `BER_pre` and/or `GCR_pre` (Pre-S5 ratings only)

**Analysis Method**:
- Non-inferiority test (candidate arms vs baseline)
- Clustered bootstrap (group-level clustering)
- Confidence intervals and Holm correction for multiple comparisons

**Data Source**: `human_ratings__arm{arm}.jsonl` → `pre_s5_rating` fields only

**Key Constraint**: **Never** use Post-S5 ratings for arm comparison.

### 5.2 Secondary Analysis (Tool Effect)

**Endpoints**: `BER_delta`, `GCR_delta`, `CT_post`

**Analysis Method**:
- Descriptive statistics (mean, SD, distribution)
- Correlation analysis: S5 flags vs human corrections
- Regression: `correction_time_ms ~ S5_blocking_flag + S5_quality_score`

**Data Source**: `human_ratings__arm{arm}.jsonl` → `post_s5_rating` and `change_log` fields

### 5.3 S5 Tool Performance Analysis (Exploratory)

**Precision@Blocking**:
- Numerator: Cards where both S5 and human flagged blocking (Pre or Post)
- Denominator: Cards where S5 flagged blocking
- Calculation: `Precision = (S5_blocking AND human_blocking) / S5_blocking`

**Recall@Blocking** (if feasible):
- Numerator: Cards where both S5 and human flagged blocking (Pre or Post)
- Denominator: Cards where human flagged blocking (Pre or Post)
- Calculation: `Recall = (S5_blocking AND human_blocking) / human_blocking`
- **Note**: May require audit sampling if Post-S5 corrections are sparse

**Analysis Output**:
- Precision/Recall metrics per arm (for consistency check)
- False positive/negative patterns
- RAG evidence quality assessment

### 5.4 Reporting Structure

**Primary Results (Arm Comparison)**:
- Table: Arm-level `BER_pre` and `GCR_pre` (mean, CI)
- Non-inferiority test results (pass/fail per candidate arm)
- Decision: Selected arm based on Pre-S5 ratings only

**Secondary Results (Tool Effect)**:
- Table: `BER_delta`, `GCR_delta`, `CT_post` (mean, SD)
- Scatter plot: S5 flags vs human corrections
- Tool performance: Precision@Blocking, Recall@Blocking (if available)

**Exploratory Results**:
- Change log analysis: Most common `change_reason_code`
- Correction time distribution
- S5 false positive/negative patterns

## 7. Guardrails & Non-Goals

### 7.1 Guardrails (Must Enforce)

#### 7.1.1 S5 Role Constraint
- **S5 is triage/flagging + evidence suggestions, NOT auto-correction**
- S5 **never** modifies upstream content (S1/S2 artifacts remain unchanged)
- S5 results are **informational only** for human raters

#### 7.1.2 No Fail-Fast Stopping
- S5 validation **never** stops the pipeline
- Pipeline continues to S6 (export) regardless of S5 results
- S5 errors are logged but do not block execution

#### 7.1.3 Arm-Independent S5 Models
- S5 models are **fixed across all arms** (same model, same configuration)
- S5 model configuration is **not** part of arm comparison
- S5 model version is logged in `s5_model_info` for reproducibility

#### 7.1.4 RAG Evidence Logging
- When S5 claims blocking error, **RAG evidence must be logged**
- RAG evidence includes: `source_id`, `source_excerpt`, `relevance`
- RAG evidence is displayed to human raters for auditability

#### 7.1.5 Pre-S5 Immutability
- Pre-S5 ratings are **immutable** after submission
- Database constraints prevent Pre-S5 field updates
- UI prevents Pre-S5 field editing after lock

#### 7.1.6 Endpoint Purity
- **Primary endpoints (arm comparison) use Pre-S5 ratings only**
- Post-S5 ratings are **never** used for arm comparison
- Analysis code must explicitly use `pre_s5_rating` fields for primary analysis

### 7.2 Non-Goals (Explicitly Excluded)

#### 7.2.1 Auto-Correction
- S5 **does not** automatically fix errors in S1/S2 content
- S5 **does not** regenerate content
- Human raters must manually correct content if needed

#### 7.2.2 Real-Time Validation
- S5 validation **does not** run during S1/S2 generation
- S5 validation runs **after** S4 (image generation) completes
- S5 results are **not** used to improve S1/S2 generation

#### 7.2.3 Multi-Arm S5 Comparison
- S5 models are **not** varied across arms
- S5 performance is **not** part of arm comparison
- S5 tool effect is measured separately (secondary endpoint)

#### 7.2.4 Heavy UI Rebuild
- **Minimal viable changes** to existing QA form
- Reuse existing UI components where possible
- Avoid multi-week UI rebuilds

## 8. Risks & Failure Modes

### 8.1 Priming Bias Risk

**Risk**: Human raters may be primed by S5 results even in Pre-S5 phase (if S5 results leak).

**Mitigation**:
- **Strict UI gating**: S5 results panel **completely hidden** until Pre-S5 submitted
- **Backend gating**: S5 reveal API endpoint verifies Pre-S5 lock exists
- **Audit trail**: Log S5 reveal timestamp to detect early access
- **Training**: Instruct raters to ignore any S5 information before Pre-S5 submission

**Detection**:
- Monitor `s5_reveal_timestamp_ms` vs `pre_lock_timestamp_ms` (should be after)
- Flag any cases where S5 revealed before Pre-S5 lock

### 8.2 Rater Fatigue Risk

**Risk**: 2-pass workflow may increase rater fatigue, leading to lower quality Post-S5 ratings.

**Mitigation**:
- **Minimal friction**: Pre-fill Post-S5 with Pre-S5 values (most cards unchanged)
- **Conditional change log**: Only require change log if fields actually changed
- **Time measurement**: Monitor correction time to detect fatigue
- **Batch design**: Limit number of cards per session

**Detection**:
- Monitor `correction_time_ms` distribution (fatigue = longer times)
- Monitor change log quality (fatigue = vague notes)

### 8.3 S5 False Positive Risk

**Risk**: S5 may flag false positives (claim blocking error when none exists), causing unnecessary corrections.

**Mitigation**:
- **RAG evidence**: Require RAG evidence when S5 claims blocking error
- **Human judgment**: Human raters can mark S5 as false positive in change log
- **Precision measurement**: Measure S5 precision@blocking to detect false positives
- **Model calibration**: Adjust S5 model threshold if false positive rate too high

**Detection**:
- Calculate S5 precision@blocking (should be >0.8)
- Monitor change log `reason_code="S5_BLOCKING_FALSE_POS"` frequency

### 8.4 S5 False Negative Risk

**Risk**: S5 may miss blocking errors (false negatives), reducing tool effectiveness.

**Mitigation**:
- **Recall measurement**: Measure S5 recall@blocking (if feasible)
- **Audit sampling**: Sample cards where human flagged but S5 did not
- **Model improvement**: Iteratively improve S5 model based on false negative patterns

**Detection**:
- Calculate S5 recall@blocking (should be >0.7)
- Audit sample: Cards where `human_blocking=true` but `S5_blocking=false`

### 8.5 Change Log Quality Risk

**Risk**: Change log entries may be vague or incomplete, reducing analysis value.

**Mitigation**:
- **Required fields**: Mandatory `change_reason_code` and `change_note` if changed
- **Validation**: Backend rejects submissions with incomplete change logs
- **Training**: Provide examples of good change log entries
- **Review**: Sample review of change log quality

**Detection**:
- Monitor change log completeness (all changes should have reason + note)
- Sample review: Check change log note quality (should be specific, not vague)

### 8.6 Endpoint Contamination Risk

**Risk**: Post-S5 ratings may accidentally be used for arm comparison, contaminating primary endpoint.

**Mitigation**:
- **Explicit field names**: Use `pre_s5_rating` and `post_s5_rating` prefixes
- **Analysis code review**: Review all analysis code to ensure Pre-S5 only
- **Documentation**: Clearly document which fields are primary endpoints
- **Validation**: Add analysis code validation to reject Post-S5 usage in primary analysis

**Detection**:
- Code review: Check all analysis scripts use `pre_s5_rating` fields
- Data validation: Flag any analysis using `post_s5_rating` for primary endpoints

### 8.7 S5 Model Version Drift Risk

**Risk**: S5 model version may change during study, causing reproducibility issues.

**Mitigation**:
- **Version locking**: Lock S5 model version at study start
- **Snapshot ID**: Include model version in `s5_snapshot_id`
- **Documentation**: Document S5 model version in study protocol
- **Validation**: Validate S5 model version consistency across all ratings

**Detection**:
- Monitor `s5_model_info` across all S5 validations (should be consistent)
- Flag any S5 model version changes during study

## 9. Decision Points

### 9.1 S5 Model Selection

**Decision**: Which models to use for S5 validation?

**Options**:
- Option A: Pro model for tables, Flash model for cards (as specified)
- Option B: Flash model for both (cost optimization)
- Option C: Pro model for both (quality optimization)

**Recommendation**: **Option A** (as specified in plan)
- **Rationale**: Tables are more complex, warrant Pro model; cards are simpler, Flash sufficient
- **Cost**: Acceptable (1 Pro call per table, 1 Flash call per card)

**Action**: Lock S5 model configuration in `S5_Validation_Contract_Canonical.md`

### 9.2 Change Log Granularity

**Decision**: How granular should change log be?

**Options**:
- Option A: One change log entry per field changed (detailed)
- Option B: One change log entry per card (aggregated)

**Recommendation**: **Option A** (one entry per field)
- **Rationale**: More detailed change log enables better analysis of S5 tool effect
- **Cost**: Minimal (most cards unchanged, so no change log needed)

**Action**: Implement change log as array of field-level changes

### 9.3 Correction Time Measurement

**Decision**: How to measure correction time?

**Options**:
- Option A: Auto-logged from timestamps (preferred)
- Option B: Self-reported time bins (fallback)

**Recommendation**: **Option A** (auto-logged)
- **Rationale**: More accurate, no user burden
- **Fallback**: Option B if timestamp logging fails

**Action**: Implement auto-logging with fallback to time bins

### 9.4 S5 Recall Measurement

**Decision**: Should we measure S5 recall@blocking?

**Options**:
- Option A: Yes, with audit sampling (recommended)
- Option B: No, precision only (simpler)

**Recommendation**: **Option A** (with audit sampling)
- **Rationale**: Recall measurement provides complete picture of S5 tool performance
- **Cost**: Additional audit sampling (manageable)

**Action**: Implement recall measurement with audit sampling plan
